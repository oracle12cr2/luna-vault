# Design Review — 기술 블로그 전문가

## 권위 기준 (Masters)

이 스킬은 3대 권위 기준을 따른다:

1. **Refactoring UI** (Adam Wathan & Steve Schoger) — 실전 UI 디자인 원칙
2. **Material Design 3** (Google) — 체계적 디자인 시스템
3. **Nielsen Norman Group** — 사용성 연구 기반 UI 가이드라인

레퍼런스 스타일: **박영민 Oracle DBA 블로그(Tistory)** — 사이드바 레이아웃, 프로필+검색+카테고리 트리 구조

> 상세 기준은 `references/masters.md` 참조.

## 🚧 앞태클 (Pre-check) — 작업 전 필수 확인

디자인 작업을 시작하기 **전에** 반드시 아래를 확인. 확인 안 되면 작업 시작하지 말 것.

1. **타겟 사용자는 누구?** — 개발자/DBA/일반 독자 등 명확히
2. **이 페이지/컴포넌트의 핵심 목표?** — 글 읽기? 탐색? 프로필 노출?
3. **기존 디자인 시스템/브랜드 가이드 있는지?** — 색상 팔레트, 폰트, 간격 규칙
4. **라이트/다크 모드 둘 다 지원하는지?**
5. **대상 뷰포트?** — 데스크톱 only? 반응형?

확인 방법:
- 사용자에게 직접 질문
- 기존 코드/CSS에서 변수/토큰 확인
- 스크린샷이 있으면 시각적으로 판단

## 📐 수치 기준 (Quantitative Checks)

AI가 자동으로 검증할 수 있는 수치 기준:

### 색상 대비 (Color Contrast)
| 항목 | 기준 | 검증 방법 |
|------|------|-----------|
| 본문 텍스트 vs 배경 | WCAG AA ≥ 4.5:1 | 대비율 계산 |
| 대형 텍스트 (18px+ bold, 24px+) vs 배경 | WCAG AA ≥ 3:1 | 대비율 계산 |
| 코드블록 텍스트 vs 코드 배경 | ≥ 4.5:1 | 대비율 계산 |
| 링크 vs 본문 텍스트 | 구분 가능한 색상 차이 (3:1 이상 또는 밑줄) | 시각적 구분 |
| 다크모드에서도 동일 기준 충족 | 모든 항목 재검증 | 다크모드 변수 확인 |

### 타이포그래피 스케일 (Typography Scale)
| 항목 | 기준 |
|------|------|
| 본문 크기 | 16px–18px (1rem–1.125rem) |
| 줄 높이 (line-height) | 본문 1.5–1.75, 제목 1.1–1.3 |
| 최대 줄 길이 | 60–80자 (영문), 한글 30–45자 (≈ 600–800px) |
| 제목 스케일 | Major Third (1.25) 또는 Perfect Fourth (1.333) 비율 |
| 폰트 종류 수 | 최대 2개 (본문 + 코드), 절대 3개 초과 금지 |
| 코드 폰트 | 모노스페이스 전용 (예: JetBrains Mono, Fira Code) |

### 여백/간격 (Spacing)
| 항목 | 기준 |
|------|------|
| 기본 간격 단위 | 4px 또는 8px 기반 그리드 |
| 섹션 간 간격 | 컴포넌트 내부 간격의 2배 이상 |
| 사이드바 너비 | 240–300px (데스크톱) |
| 콘텐츠 영역 최대 너비 | 680–800px |
| 패딩 일관성 | 같은 레벨 요소는 같은 패딩 |

### 코드블록
| 항목 | 기준 |
|------|------|
| 코드 폰트 크기 | 본문의 85–90% (14px–15px) |
| 코드블록 배경 | 본문 배경과 구분되는 색 (다크모드: 더 어둡게, 라이트: 더 밝게) |
| 패딩 | 최소 16px |
| border-radius | 4–8px |
| 가로 스크롤 | overflow-x: auto (줄바꿈 금지) |
| 구문 강조 | 지원 필수 |

## 👁️ 사람 눈 기준 (Visual Scenarios)

수치로 잡을 수 없는 항목. 시나리오 기반으로 판단:

### 시각적 계층 (Visual Hierarchy)
- 페이지를 5초간 봤을 때 **제목 → 본문 → 사이드바** 순서로 시선이 가는가?
- 제목이 본문보다 확실히 눈에 띄는가? (크기, 굵기, 색상 차이)
- CTA나 중요 요소가 시각적으로 부각되는가?

### 코드블록 가독성
- 코드가 본문과 명확히 구분되는가?
- 구문 강조 색상이 배경과 충분히 대비되는가?
- 긴 코드에서 가로 스크롤이 자연스러운가?

### 다크모드 일관성
- 라이트↔다크 전환 시 **모든** 요소가 적절히 변환되는가?
- 그림자/테두리가 다크모드에서도 자연스러운가?
- 이미지/아이콘이 다크 배경에서 사라지지 않는가?

### 사이드바 레이아웃 (Tistory 스타일)
- 프로필 영역이 상단에 깔끔하게 배치되는가?
- 검색창이 즉시 보이는가?
- 카테고리 트리가 접기/펼치기 가능하고 현재 위치가 표시되는가?
- 사이드바와 콘텐츠 사이 간격이 충분한가?

## 🚫 안티패턴 (절대 하면 안 되는 것)

1. **폰트 3개 초과** — 본문 + 코드 + (선택적 제목) 최대 3개
2. **과도한 애니메이션** — 읽기 흐름을 방해하는 모든 움직임 (parallax, 자동 슬라이드 등)
3. **읽기 흐름 방해** — 콘텐츠 중간에 끼어드는 광고/배너/팝업
4. **불일치 간격** — 같은 레벨 요소에 다른 패딩/마진
5. **색상 남용** — 의미 없는 색 사용, 무지개 UI
6. **코드블록 줄바꿈** — 코드는 반드시 가로 스크롤 (word-wrap 금지)
7. **다크모드 미적용 요소** — 흰 배경 이미지, 하드코딩된 색상
8. **너무 작은 텍스트** — 본문 14px 미만 절대 금지
9. **콘텐츠 영역 너무 넓음** — max-width 없이 전체 화면 사용
10. **사이드바 스크롤** — 사이드바가 뷰포트보다 길어서 별도 스크롤 필요

## ✅ 백태클 (Post-validation) — 작업 후 자체 검증

작업 완료 후 반드시 아래 루프를 실행:

### Step 1: 수치 기준 자동 검증
```
□ 색상 대비율 전체 항목 (본문, 제목, 코드, 링크) — 라이트/다크 모두
□ 타이포 스케일 비율 확인
□ 줄 길이 60–80자 범위
□ 간격 그리드 일관성 (4px/8px)
□ 코드블록 스펙 (폰트, 크기, 패딩, overflow)
□ 폰트 종류 ≤ 3개
□ 콘텐츠 영역 max-width 설정됨
□ 사이드바 너비 240–300px
```

### Step 2: 미달 시 자동 수정
- 미달 항목 발견 → **사용자에게 보고하지 말고** 즉시 수정
- 수정 후 Step 1 재실행
- 모든 항목 통과할 때까지 반복

### Step 3: 사람 눈 기준 보고
모든 수치 기준 통과 후에만 아래 시나리오 결과를 사용자에게 보고:
- 시각적 계층 평가 결과
- 코드블록 가독성 평가
- 다크모드 일관성 평가
- 사이드바 레이아웃 평가
- 안티패턴 위반 여부

### 보고 형식
```
## 디자인 리뷰 결과

### 수치 기준 (전체 통과 ✅)
- 색상 대비: ✅ (본문 7.2:1, 코드 5.8:1, ...)
- 타이포 스케일: ✅ (1.25 비율)
- ...

### 시각적 평가
- 계층: [평가]
- 코드블록: [평가]
- 다크모드: [평가]
- 사이드바: [평가]

### 개선 제안 (있을 경우)
1. ...
```
