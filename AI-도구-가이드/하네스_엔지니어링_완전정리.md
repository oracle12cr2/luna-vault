## 1. 하네스 엔지니어링이란?

AI 에이전트가 **신뢰성 있게 작업을 수행할 수 있도록, 에이전트 바깥의 환경(도구·규칙·문서·피드백 루프)을 설계하는 엔지니어링 분야**.

"하네스(harness)"는 말의 고삐와 안장 — 말(에이전트)이 올바른 방향으로 힘을 쓰도록 제어하는 장치.

> "에이전트가 실수를 하면, 에이전트가 그 실수를 **다시는 하지 못하도록** 환경을 엔지니어링하라."
> — Mitchell Hashimoto

---

## 2. 진화 흐름: 프롬프트 → 컨텍스트 → 하네스

| 시기 | 단계 | 핵심 | 비유 |
|------|------|------|------|
| 2023~2024 | **프롬프트 엔지니어링** | 질문을 잘 쓴다 | 주문서를 잘 쓰기 |
| 2025 | **컨텍스트 엔지니어링** | RAG, MCP, 메모리 등 맥락을 설계한다 | 등산 전 지도·경로를 주기 |
| **2026** | **하네스 엔지니어링** | 실수해도 자동 복구되는 환경을 만든다 | 등산 중 안전 로프 |

### 컨텍스트 vs 하네스 비교

| 구분 | 컨텍스트 엔지니어링 | 하네스 엔지니어링 |
|------|---------------------|-------------------|
| **핵심 역할** | 큰 방향성 조정 | 짧은 주기의 자동 오류 복구 |
| **시간 축** | 태스크 시작 시점 | 매 실행 루프 |
| **설계 대상** | LLM이 보는 모든 토큰 | 에이전트 바깥의 제약·피드백 시스템 |
| **비유** | 지도 없으면 어디로 가야 할지 모름 | 안전 로프 없으면 한 번 실수에 추락 |

> "컨텍스트 엔지니어링은 모델이 잘 **생각**하게 돕고, 하네스 엔지니어링은 시스템이 **궤도를 이탈하지 않게** 막는다." — Martin Fowler

## 3. 하네스의 구성 요소

| 구성 요소 | 설명 | 예시 |
|-----------|------|------|
| **저장소 구조** | 디렉토리 컨벤션, 파일 명명 규칙 | 모노레포, 도메인별 레이어 분리 |
| **CI/CD 파이프라인** | 자동 테스트, 빌드, 배포 | 매 커밋마다 테스트 실행 |
| **린터/포매터** | 코드 스타일의 기계적 강제 | ESLint, 커스텀 린트 규칙 |
| **아키텍처 제약** | 구조적 테스트, 의존성 방향 검증 | 레이어 간 의존성 린트 |
| **프로젝트 지시 파일** | 에이전트용 지침서 | AGENTS.md, CLAUDE.md, .cursorrules |
| **지식 베이스** | 구조화된 문서, 설계 히스토리 | docs/ 디렉토리, 아키텍처 문서 |
| **툴 연결** | 외부 도구 접근 | MCP 서버, 브라우저 자동화, 로그 조회 |
| **권한/승인 흐름** | 에이전트 행동 범위 제한 | 파일 수정 권한, 배포 승인 |
| **실행 루프** | 재시도, 에러 처리, 타임아웃 | 실패 시 자동 재시도, 롤백 |
| **엔트로피 관리** | 코드베이스 일관성 주기적 점검 | doc-gardening 에이전트, GC 에이전트 |

## 4. OpenAI가 증명한 것 — Codex 실험

### 실험 개요

- **기간**: 5개월 (2025년 8월~)
- **규모**: 3명 엔지니어 → 7명으로 확장
- **결과**: **100만 줄+ 프로덕션 코드**, 수동 작성 코드 **0줄**
- **처리량**: 엔지니어 1인당 하루 평균 3.5 PR, 총 1,500+ PR
- **추정**: 수동 대비 **1/10 시간**으로 완성

### 핵심 교훈

> "엔지니어들이 한 일은 코드를 짜는 게 아니었다. **하네스를 설계하는 것**이었다."

#### OpenAI의 하네스 3대 축

**① 컨텍스트 엔지니어링**

- AGENTS.md는 백과사전이 아닌 **목차(table of contents)** 로 사용 (~100줄)
- 실제 지식은 구조화된 `docs/` 디렉토리에 보관
- **점진적 공개(Progressive Disclosure)**: 작은 진입점에서 시작 → 필요할 때 더 깊이 탐색
- 에이전트가 접근할 수 없는 정보는 존재하지 않는 것과 같음 (Slack 대화, Google Docs ❌)
- 모든 맥락을 **레포지토리 내 버전 관리되는 아티팩트**로 관리

```
docs/
├── architecture/     ← 도메인 맵, 패키지 레이어링
├── designs/          ← 설계 문서 (검증 상태 포함)
├── plans/            ← 실행 계획, 진행 로그, 의사결정 기록
├── quality/          ← 도메인별 품질 등급, 기술 부채 추적
└── beliefs/          ← 에이전트 우선 운영 원칙
```

**② 아키텍처 제약 (기계적 강제)**

- 비즈니스 도메인별 고정된 레이어 구조 강제:
  ```
  Types → Config → Repo → Service → Runtime → UI
  ```
- 의존성 방향은 단방향만 허용, 커스텀 린터로 기계적 검증
- 린트 에러 메시지에 **수정 방법(remediation)을 포함** → 에이전트가 바로 고침
- 구조화된 로깅, 네이밍 컨벤션, 파일 크기 제한 등 정적 검증

> "인간 워크플로우에서는 이 규칙들이 지나치게 느껴질 수 있다. 에이전트에게는 **곱하기 효과(multiplier)**가 된다."

**③ 가비지 컬렉션 (엔트로피 관리)**

- **doc-gardening 에이전트**: 주기적으로 문서-코드 불일치 탐지 → 자동 수정 PR
- CI 잡으로 지식 베이스 최신 상태·교차 링크·구조 검증
- 기술 부채도 버전 관리, 능동적으로 추적

#### 기타 핵심 실천법

- **에이전트-to-에이전트 리뷰**: 인간 리뷰 최소화, 에이전트가 자체 리뷰 + 상호 리뷰
- **관찰 도구 접근**: 로그(LogQL), 메트릭(PromQL), 트레이스를 에이전트에 노출
- **브라우저 자동화**: Chrome DevTools Protocol로 DOM 스냅샷, 스크린샷, UI 검증
- **Git worktree 격리**: 변경 건별 독립된 앱 인스턴스 + 관찰 스택
- **단일 Codex 실행이 6시간+ 작업** (엔지니어가 자는 동안 진행)
- **"지루한(boring)" 기술 선호**: 안정적이고 구성 가능한 기술이 에이전트에 유리

## 5. Anthropic이 증명한 것 — Long-Running Agent

### 핵심 문제

Long-running agent는 여러 컨텍스트 윈도우를 넘나들며 작업하는데, **매 세션이 이전 기억 없이 시작됨** — "전 교대 기억 없이 출근하는 엔지니어"

### 2가지 실패 패턴

| 패턴 | 설명 |
|------|------|
| **한 번에 다 하려는 문제** | 전체 앱을 원샷으로 구현 시도 → 컨텍스트 소진 → 반쯤 구현된 상태로 남김 |
| **조기 완료 선언** | 이미 된 부분 보고 "끝났다"고 판단 → 미완성 기능 방치 |

### 해결: 2중 에이전트 구조

#### ① Initializer Agent (첫 세션)

첫 실행 시 환경 세팅:

- `init.sh` — 개발 서버 실행 스크립트
- `claude-progress.txt` — 진행 상황 로그 파일
- `feature_list.json` — 200+ 기능 목록 (모두 `"passes": false`로 시작)
- 초기 Git 커밋

```json
{
  "category": "functional",
  "description": "New chat button creates a fresh conversation",
  "steps": [
    "Navigate to main interface",
    "Click the 'New Chat' button",
    "Verify a new conversation is created"
  ],
  "passes": false
}
```

> 💡 **JSON 사용 이유**: Markdown보다 모델이 부적절하게 변경할 확률이 낮음

#### ② Coding Agent (이후 모든 세션)

매 세션 시작 시:

```
1. pwd → 작업 디렉토리 확인
2. claude-progress.txt 읽기 → 이전 작업 파악
3. feature_list.json 읽기 → 다음 작업 선택
4. git log → 최근 변경 확인
5. init.sh 실행 → 개발 서버 시작
6. 기본 기능 테스트 → 앱이 깨졌는지 확인
7. 단일 기능 구현 → 한 번에 하나만!
8. 브라우저 자동화로 E2E 테스트
9. Git 커밋 + 진행 상황 기록 → 다음 세션을 위해
```

### 실패 패턴별 해결책

| 문제 | Initializer 해결 | Coding Agent 해결 |
|------|-----------------|------------------|
| 조기 완료 선언 | feature_list.json 설정 | 세션 시작 시 feature 목록 읽기 |
| 환경이 깨진 상태로 남김 | 초기 Git repo + 진행 파일 | 세션 시작 시 기본 테스트, 종료 시 커밋 |
| 기능 미검증 완료 표시 | feature_list 구조화 | 브라우저 자동화로 E2E 검증 후에만 passes=true |
| 앱 실행 방법 모름 | init.sh 스크립트 | 세션 시작 시 init.sh 읽기 |

## 6. 4가지 핵심 원칙

### ① 피드백 루프를 짧게, 자동 복구를 빠르게

- 5분 차에 실수 → 5분 차에 린터가 잡음 → 바로 수정
- 30분 뒤에 발견하면 이미 그 위에 코드가 쌓여 복구 비용 급증
- **매 실행 루프마다** 검증·복구가 핵심

### ② 희망 대신 기계적 강제

- "이렇게 해줘" (희망) ❌
- "그렇게 안 하면 빌드가 실패하게" (기계적 강제) ✅
- 린터, CI, 구조적 테스트로 규칙을 코드화

### ③ 엔트로피와 싸워라

- 에이전트가 많은 코드를 생성할수록 일관성 저하
- 주기적으로 코드베이스 점검하는 GC(가비지 컬렉션) 에이전트
- 문서-코드 불일치 자동 탐지 → 수정 PR

### ④ 컨텍스트로 방향, 하네스로 걸음을 지켜라

- 컨텍스트만 있으면: 방향은 알지만 매번 넘어짐
- 하네스만 있으면: 넘어지진 않지만 어디로 가야 할지 모름
- **둘 다 있어야** 긴 태스크를 완주

## 7. 실전 체크리스트

### 기본 하네스 (모든 프로젝트)

- [ ] **AGENTS.md / CLAUDE.md** — 목차 역할 (~100줄), 상세는 docs/에 분리
- [ ] **CI/CD** — 매 커밋마다 자동 테스트
- [ ] **린터/포매터** — 코드 스타일 기계적 강제
- [ ] **Git 규율** — 의미 있는 커밋 메시지, 기능 단위 PR
- [ ] **진행 상황 파일** — 에이전트 간 인수인계용 (progress.txt / progress.json)

### 중급 하네스 (복잡한 프로젝트)

- [ ] **커스텀 린트 규칙** — 아키텍처 제약을 코드로 검증
- [ ] **구조화된 문서** — docs/ 디렉토리, 설계 문서, 아키텍처 맵
- [ ] **에이전트 리뷰** — 에이전트가 자체 코드를 리뷰
- [ ] **브라우저/UI 자동화** — E2E 테스트
- [ ] **관찰 도구 접근** — 로그, 메트릭을 에이전트에 노출

### 고급 하네스 (프로덕션 수준)

- [ ] **도메인별 레이어 구조** — 의존성 방향 기계적 검증
- [ ] **GC 에이전트** — 문서 불일치, 아키텍처 위반 주기적 점검
- [ ] **멀티 에이전트** — 코딩/테스팅/QA/정리 전문 에이전트 분리
- [ ] **점진적 공개** — 작은 진입점 → 필요 시 깊은 컨텍스트
- [ ] **실행 계획 관리** — 계획·진행·의사결정 모두 버전 관리
