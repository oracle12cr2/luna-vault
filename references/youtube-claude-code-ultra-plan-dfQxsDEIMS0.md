## 한줄 요약

Claude Code를 단순 채팅 도구가 아니라, 여러 역할 에이전트를 병렬로 돌려 기획/탐색/비평을 수행하는 작업 오케스트레이션 도구로 쓰는 방법을 설명하는 영상이다.

## 영상 핵심 포인트

### 1. 병렬 에이전트 구조

영상에서는 예시로 다음과 같은 구성을 말한다.

- 탐색 에이전트 3명
- 비평 에이전트 1명

이렇게 여러 역할을 동시에 돌려서
- 아이디어 탐색
- 기획 검토
- 품질 비평
을 병렬 처리한다.

## 2. 기대 효과

- 순차 작업보다 속도 단축
- 다양한 관점 확보
- 품질 검토 강화
- 사용자는 중간에 다른 작업을 할 수 있음

영상에서는 대략 15분 걸리던 작업이 5분 정도로 줄어드는 식의 체감 이점을 강조한다.

## 3. Claude Code 사용 방식

Claude Code를 다음 두 흐름으로 설명한다.

- 터미널에서 직접 실행
- VS Code 확장으로 실행

즉 Claude Code를 단순 콘솔형 도구가 아니라 IDE 기반 작업 허브처럼 활용하는 관점이다.

## 4. Plan Mode의 의미

영상에서 강조하는 핵심은 실제 편집 전에 먼저 계획을 세우는 것이다.

관련 모드 예시:
- Plan Mode
- Ask Before Edit
- Edit Automatically
- Bypass Permission

Plan Mode는 실제 수정 전에
- 무엇을 할지
- 어떤 절차로 진행할지
- 준비가 됐는지
를 점검하는 단계로 설명된다.

## 5. 비평 에이전트의 역할

탐색/실행 에이전트만 두는 게 아니라, 별도의 비평 에이전트를 둬서
- 기획의 허점
- 구현 준비 상태
- 리스크
를 점검하게 하는 구조가 포인트다.

이건 단순 병렬화가 아니라 "병렬 + 검증" 구조다.

## 우리 workflow 관점 해석

이 영상은 단순 사용 팁보다 다음 운영 원칙과 잘 맞는다.

### A. 계획 먼저, 실행 나중
- 바로 코딩하지 않고 먼저 plan을 세움
- 우리 쪽의 plan/update_plan, 설계 먼저 보기와 결이 맞음

### B. 역할 분리
- 탐색 역할과 비평 역할을 분리
- 우리 쪽의 subagent / Codex review / Claude planning 역할 분리와 유사함

### C. 병렬화 + 검증
- 여러 에이전트를 동시에 돌리되, 비평/리뷰 단계를 둠
- 단순 속도 향상보다 품질 관리가 중요하다는 점이 핵심

## 우리에게 적용할 만한 포인트

1. 큰 작업은 planner/reviewer 역할을 먼저 분리
2. 구현 에이전트와 비평 에이전트를 따로 둠
3. 편집 전 Plan Mode 성격의 초안을 먼저 만들고 시작
4. 병렬화는 무조건 많이보다, 역할이 다른 작업만 병렬화

## 주의할 점

- 병렬 에이전트가 많아질수록 컨텍스트 관리가 어려워짐
- 계획 없이 병렬화만 하면 중복/충돌 가능성이 큼
- 비평 에이전트가 없으면 속도만 빨라지고 품질은 흔들릴 수 있음

## 결론

이 영상의 본질은 Claude Code의 특정 기능 소개라기보다, AI 에이전트를 "기획 → 탐색 → 비평 → 실행" 구조로 운영하는 방식 제안에 가깝다.

우리 기준에서는 Claude Code에만 한정하지 말고, OpenClaw + Codex + Claude 조합에서도 같은 원칙으로 응용할 수 있다.
